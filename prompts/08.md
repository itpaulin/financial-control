# Contexto

O projeto hoje é 100% client-side (Zustand + localStorage). A stack declarada em `AGENTS.md` e nas cursor rules exige Prisma 7 + Better Auth, mas nada está instalado. Esta etapa introduz autenticação e banco de dados — mas **somente a camada de auth**. Migrar os dados do Zustand pro DB fica pra próxima etapa.

## Tarefa

Implementar Auth + DB usando **Prisma 7** + **Better Auth** + **Neon** (serverless Postgres, free tier, integra com Vercel).

Nesta primeira etapa:
- Login e cadastro simples por email e senha.
- Proteger **todas** as rotas existentes (middleware + layout server-side).
- Sem social login, 2FA, magic link ou reset de senha.

## Decisões

- **DB**: Neon (serverless Postgres). Driver adapter `@prisma/adapter-neon` + `@neondatabase/serverless`.
- **Sign-up**: tela pública em `/sign-up`.
- **Proteção**: defesa em camadas — `middleware.ts` (redirect otimista via cookie, edge-safe) + layout server-side (valida sessão com `auth.api.getSession`).

## Arquivos-chave

- `prisma.config.ts`, `prisma/schema.prisma` (modelos gerados pelo Better Auth CLI)
- `lib/prisma.ts` — singleton do PrismaClient com Neon adapter
- `lib/auth.ts` — config do Better Auth (emailAndPassword + nextCookies plugin)
- `lib/auth-client.ts` — cliente pra uso em Client Components
- `app/api/auth/[...all]/route.ts` — catch-all handler
- `app/(auth)/sign-in/page.tsx` e `app/(auth)/sign-up/page.tsx`
- `app/(protected)/layout.tsx` — redirect se sem sessão
- `middleware.ts` — `getSessionCookie` de `better-auth/cookies`

## Variáveis de ambiente

```
DATABASE_URL="postgresql://...neon.tech/...?sslmode=require"
BETTER_AUTH_SECRET="..."           # openssl rand -base64 32
BETTER_AUTH_URL="http://localhost:3000"
```

## Verificação

- Acessar `/` sem cookie → redireciona pra `/sign-in`.
- Cadastrar em `/sign-up` → autoSignIn leva pra `/`.
- Registros aparecem no Neon console (User, Session, Account).
- Logout limpa cookie e redireciona.
- `bunx tsc --noEmit` e `bun run lint` sem erros.
